[0] EXECUTIVE SUMMARY
================================================================================

수능 영어 문항 출제를 위한 end-to-end 워크벤치.
학술 소스에서 지문을 검색/추출 -> 적합성 평가 -> 문항 생성 -> 해설 작성의
전체 파이프라인을 하나의 대시보드에서 수행한다.

핵심 설계 원칙:
  1. Verbatim-only: 원문에서 한 글자도 변형 없이 추출. 환각 원천 차단.
  2. Human-in-the-loop: 모든 단계에서 사용자 피드백 -> 모델 품질 점진 향상.
  3. Engine 분리: 검색/평가 | 문항생성 | 해설작성 을 독립 엔진으로 설계.
  4. Claude API 우선: RunPod/HuggingFace는 fine-tuning 필요시에만 사용.
  5. Seed-first: 실제 평가원 기출 데이터로 cold start 해결 후 피드백으로 점진 개선.

--- 3.2 Verbatim 보장 메커니즘 ---

  핵심: Claude에게 "생성"이 아닌 "위치 지정"만 시킴.

  방법:
  1. 원문 전문을 Claude에게 전달
  2. Claude는 시작 문장과 끝 문장의 위치(index)만 반환
  3. 백엔드가 해당 위치를 기반으로 원문에서 직접 슬라이싱
  4. 슬라이싱 결과를 다시 원문과 대조하여 100% 일치 검증
  5. 불일치 시 reject 처리

  Claude Tool Schema:

  {
    "name": "extract_passage",
    "description": "원문에서 수능 지문으로 적합한 구간의 위치를 지정",
    "input_schema": {
      "type": "object",
      "properties": {
        "start_sentence_index": {
          "type": "integer",
          "description": "추출 시작 문장의 인덱스 (0-based)"
        },
        "end_sentence_index": {
          "type": "integer",
          "description": "추출 종료 문장의 인덱스 (inclusive)"
        },
        "reasoning": {
          "type": "string",
          "description": "이 구간을 선택한 이유"
        }
      },
      "required": ["start_sentence_index", "end_sentence_index", "reasoning"]
    }
  }


--- 3.3 Word Count Tool (Function Calling) ---

  {
    "name": "count_words",
    "description": "주어진 영문 텍스트의 정확한 단어 수를 반환",
    "input_schema": {
      "type": "object",
      "properties": {
        "text": {
          "type": "string",
          "description": "단어 수를 셀 텍스트"
        }
      },
      "required": ["text"]
    }
  }

  *실제 단어 수 계산은 백엔드에서 수행 (split by whitespace).
  Claude가 단어를 "세는" 것이 아니라, tool을 호출하여 정확한 값을 받음.
  155~180 범위 밖이면 구간 재조정 요청.


--- 3.4 Quality Scoring System ---

  점수 체계 (각 0~10, 총 100점 환산):

  Metric                  | Weight | Description
  ------------------------+--------+--------------------------------------------
  topic_depth             |  15%   | 주제의 깊이/학술성 (너무 쉽거나 어렵지 않은)
  logical_structure       |  20%   | 논리 전개의 명확성 (주장-근거-결론)
  standalone_coherence    |  15%   | 발췌만으로 의미가 완결되는 정도
  vocabulary_level        |  15%   | 수능 수준에 적합한 어휘 난이도
  question_type_fit       |  20%   | 특정 문항 유형에 얼마나 잘 맞는지
  distractor_potential    |  15%   | 매력적인 오답 선지 생성 가능성

  유형 태깅 (다중 선택 가능, 12개 유형):
  - 20번 주장 (claim)
  - 21번 함축 의미 (implication)
  - 22번 요지 (gist)
  - 23번 주제 (topic)
  - 24번 제목 (title)
  - 29번 어법 (grammar)
  - 30번 어휘 (vocabulary)
  - 31~34번 빈칸 추론 (blank)
  - 35번 무관한 문장 (irrelevant)
  - 36~37번 순서 배열 (order)
  - 38~39번 문장 삽입 (insertion)
  - 40번 요약문 (summary)
  * 41~42번 장문A는 title + vocabulary 복합 (jangmun_title, jangmun_vocabulary)

  사용자 워크플로우:
  1. Claude가 각 metric 점수 + 유형 태그를 자동 산출
  2. Pane 2에 점수와 태그 표시
  3. 사용자가 점수/태그를 수정 가능 (인라인 편집)
  4. 수정된 값은 feedback DB에 저장
  5. 향후 평가 시 최근 N개 피드백을 few-shot example로 주입

--- 4.0 Seed Data: 평가원 기출 기반 초기 학습 데이터 ---

  *** Cold Start 문제 ***
  피드백이 0건인 초기 상태에서 Claude는 "수능 지문다움"의 기준이 없음.
  실제 평가원 기출 문항을 Golden Dataset으로 제공하여 해결.

  *** 데이터 파일 구조 ***
  seed-data/
  ├── passages/           # 유형별 기출 지문 (JSONL 포맷)
  │   ├── blank.jsonl     # 빈칸 추론
  │   ├── order.jsonl     # 순서 배열
  │   ├── insertion.jsonl # 문장 삽입
  │   ├── topic.jsonl     # 주제/제목
  │   ├── main_idea.jsonl # 요지/주장
  │   ├── vocabulary.jsonl# 어휘
  │   └── irrelevant.jsonl# 무관한 문장
  └── negative/
      └── rejected.jsonl  # 부적합 예시 (경계 사례 학습용)

  *** JSONL 레코드 스키마 (실제 데이터 기준) ***
  {
    "question_code": 230631,                   // YYMMQQ (연도+월+문항번호)
    "question_number": 31,                     // 문항 번호
    "question_type": "blank",                  // 유형 코드 (12종)
    "passage": "지문 본문 + 선지 포함 텍스트",   // 시험지 원문 그대로
    "word_count": 171,                         // 지문 단어 수
    "answer": 2,                               // 정답 번호
    "jangmun_set": true,                       // (선택) 장문 세트 여부
    "jangmun_note": "장문 독해 - 제목 (41번)"   // (선택) 장문 메모
  }

  유형 코드 (12종):
    claim | implication | gist | topic | title | grammar |
    vocabulary | blank | irrelevant | order | insertion | summary

  *** 출제 지침 데이터 (guidelines.json) ***
  seed-data/guidelines.json에 유형별 출제 규칙을 구조화하여 저장.
  - global: 모든 유형 공통 대전제 (단어수, 원문 훼손 지양 등)
  - types.{type}: 유형별 passage_criteria, choice_rules, pitfalls
  - inherits: 장문 유형은 일반 유형 규칙을 상속
  이 데이터는 Claude system prompt에 해당 유형 규칙으로 주입됨.

  *** 실제 데이터 현황 (228개 기출) ***
  - blank: 48개 | order: 24개 | insertion: 24개 | title: 24개
  - vocabulary: 24개 | topic: 12개 | claim: 12개 | gist: 12개
  - implication: 12개 | grammar: 12개 | summary: 12개 | irrelevant: 12개

  *** 시스템 로드 방식 ***

  [서버 시작 / seed 변경 감지]
       |
       v
  [Seed Loader] (src/lib/seed-loader.ts)
       |
       | 1. seed-data/**/*.jsonl 파일 전부 파싱
       | 2. 각 레코드를 SeedExample 타입으로 변환
       | 3. DB seed_examples 테이블에 upsert (중복 방지)
       | 4. 각 지문의 embedding 생성 (pgvector)

  [Prompt Builder에서 활용] (아래 4.1에서 상세)

--- 4.1 전략: Seed + Few-shot Retrieval (3단계 진화) ---

  왜 이 방식이 최적인가:
  - Claude API는 fine-tuning을 지원하지 않음
  - 하지만 충분한 few-shot examples + system prompt로 동등한 효과 달성 가능
  - 피드백이 즉시 반영됨 (학습 사이클 없음)
  - 비용 효율적 (GPU 불필요)

  *** 3단계 진화 모델 ***

  Stage 1 - Seed Only (Day 1~)
  ┌─────────────────────────────────────────────────┐
  │  System Prompt                                  │
  │  ├── 역할 정의 + 평가 기준                       │
  │  ├── [이상적인 지문 예시] ← seed data에서 로드   │
  │  │     유형별 승인 예시 3~5개                     │
  │  │     부적합 예시 2~3개                          │
  │  └── [절대 규칙] verbatim 등                     │
  └─────────────────────────────────────────────────┘
  - 사용자 피드백 없이 seed data만으로 운영
  - 이 단계에서도 유의미한 지문 선별 품질 확보

  Stage 2 - Seed + Feedback (피드백 축적 시작 후)
  ┌─────────────────────────────────────────────────┐
  │  System Prompt                                  │
  │  ├── 역할 정의 + 평가 기준                       │
  │  ├── [이상적인 지문 예시] ← seed data            │
  │  │     (기출 기반, 변하지 않는 골든 기준)          │
  │  ├── [사용자 피드백 예시] ← feedback DB           │
  │  │     pgvector 유사도로 검색한 최근 피드백        │
  │  │     승인 예시 + 거부 예시                      │
  │  └── [절대 규칙]                                 │
  └─────────────────────────────────────────────────┘
  - Seed = 불변의 기준선 (anchor)
  - Feedback = 사용자 취향/미세 조정 반영
  - 두 소스 모두 활용하되, 최근 피드백에 가중치

  Stage 3 - Feedback Dominant (피드백 100+개 후)
  ┌─────────────────────────────────────────────────┐
  │  System Prompt                                  │
  │  ├── 역할 정의 + 평가 기준                       │
  │  ├── [기준 예시] ← seed 핵심 3개만 유지           │
  │  ├── [학습된 패턴] ← feedback DB                  │
  │  │     유사도 Top-K + 최근 피드백 혼합             │
  │  │     (context window 제한 내에서 최적 배분)      │
  │  └── [절대 규칙]                                 │
  └─────────────────────────────────────────────────┘
  - Seed는 anchor 역할만 (최소화)
  - 대부분의 few-shot 슬롯을 실제 피드백이 차지
  - 사용자의 고유한 평가 기준이 모델에 체화됨

  *** Example Selection 알고리즘 ***

  def select_examples(query_passage, question_type, stage):
      MAX_EXAMPLES = 10  # context window budget

      # 항상 seed에서 기본 예시 확보
      seed_approved = get_seed_by_type(question_type, approved=True)
      seed_rejected = get_seed_by_type(question_type, approved=False)

      if stage == 1:  # seed only
          return seed_approved[:5] + seed_rejected[:3]

      # feedback DB에서 유사 예시 검색
      similar_feedback = pgvector_search(
          query=embed(query_passage),
          filter_type=question_type,
          limit=MAX_EXAMPLES
      )

      if stage == 2:  # seed + feedback
          seed_slots = 4
          feedback_slots = MAX_EXAMPLES - seed_slots
          return (seed_approved[:2] + seed_rejected[:2] +
                  similar_feedback[:feedback_slots])

      if stage == 3:  # feedback dominant
          seed_slots = 2
          feedback_slots = MAX_EXAMPLES - seed_slots
          return (seed_approved[:1] + seed_rejected[:1] +
                  similar_feedback[:feedback_slots])

  *** Feedback DB 저장 구조 ***

  [사용자 피드백 입력]
       |
       v
  [Feedback DB 저장]
  {
    passage_id, passage_text, source_url, word_count,
    ai_scores: {topic_depth: 7, ...},
    user_scores: {topic_depth: 9, ...},   // 사용자 수정값
    user_comment: "논리 전개가 매우 좋음",
    question_types: ["blank", "order"],
    approved: true/false,
    created_at: timestamp
  }
       |
       v
  [다음 평가 요청 시]
       |
       +-> pgvector로 유사한 과거 피드백 Top-K 검색
       +-> Seed data와 합쳐서 System prompt에 삽입
       |
       v
  [Claude가 학습된 기준으로 평가]

--- 4.2 향후 확장: RunPod Fine-tuning (Phase 3, Optional) ---

  피드백 데이터가 충분히 쌓이면 (500+ examples):

  1. Seed data + Feedback DB에서 학습 데이터 export
  2. HuggingFace 모델 선택 (추천: Mistral-7B 또는 Llama-3-8B)
  3. RunPod에서 LoRA fine-tuning 실행
  4. Fine-tuned 모델을 RunPod Serverless로 배포
  5. 평가 엔진에서 Claude 대신 (또는 병행) 사용

  이 전환은 비용 절감 + 일관성 향상이 검증될 때만 진행.
  초기에는 Claude API + seed/feedback few-shot만으로 충분.

  [Pane 2에서 승인된 지문들]

-- Seed Data (평가원 기출 골든 데이터셋)
CREATE TABLE seed_examples (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    question_code   INTEGER NOT NULL UNIQUE,  -- YYMMQQ (예: 230631)
    question_number SMALLINT NOT NULL,
    question_type   TEXT NOT NULL,             -- 12종 유형 코드
    passage         TEXT NOT NULL,             -- 지문 + 선지 포함 원문
    passage_only    TEXT,                      -- 선지 분리된 순수 지문
    choices         JSONB,                     -- 분리된 선지 배열
    word_count      INTEGER NOT NULL,
    answer          SMALLINT NOT NULL,         -- 정답 번호
    is_jangmun      BOOLEAN DEFAULT false,    -- 장문 세트 여부
    jangmun_note    TEXT,                      -- 장문 메모
    embedding       vector(1536),
    created_at      TIMESTAMPTZ DEFAULT now()
);

-- 원문 소스
CREATE TABLE sources (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    title           TEXT NOT NULL,
    authors         TEXT[],
    source_url      TEXT NOT NULL,
    repository      TEXT NOT NULL,          -- 'arxiv', 'semantic_scholar', etc.
    full_text       TEXT,                   -- 원문 전체
    fetched_at      TIMESTAMPTZ DEFAULT now()
);

-- 추출된 지문
CREATE TABLE passages (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    source_id       UUID REFERENCES sources(id),
    text            TEXT NOT NULL,           -- verbatim 추출된 지문
    start_index     INTEGER NOT NULL,        -- 원문 내 시작 문장 인덱스
    end_index       INTEGER NOT NULL,        -- 원문 내 종료 문장 인덱스
    word_count      INTEGER NOT NULL,
    embedding       vector(1536),            -- pgvector for similarity search
    status          TEXT DEFAULT 'pending',  -- pending | approved | rejected
    created_at      TIMESTAMPTZ DEFAULT now()
);

-- 지문 점수 (AI 산출 + 사용자 수정)
CREATE TABLE passage_scores (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    passage_id      UUID REFERENCES passages(id),
    scorer          TEXT NOT NULL,           -- 'ai' | 'user'
    topic_depth     SMALLINT CHECK (topic_depth BETWEEN 0 AND 10),
    logical_structure SMALLINT CHECK (logical_structure BETWEEN 0 AND 10),
    standalone_coherence SMALLINT CHECK (standalone_coherence BETWEEN 0 AND 10),
    vocabulary_level SMALLINT CHECK (vocabulary_level BETWEEN 0 AND 10),
    question_type_fit SMALLINT CHECK (question_type_fit BETWEEN 0 AND 10),
    distractor_potential SMALLINT CHECK (distractor_potential BETWEEN 0 AND 10),
    total_weighted  DECIMAL(5,2),           -- 가중 합산 점수
    question_types  TEXT[],                 -- ['blank', 'order', ...]
    comment         TEXT,
    created_at      TIMESTAMPTZ DEFAULT now()
);

-- 생성된 문항
CREATE TABLE questions (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    passage_id      UUID REFERENCES passages(id),
    question_type   TEXT NOT NULL,
    question_text   TEXT NOT NULL,
    passage_modified TEXT NOT NULL,
    choices         JSONB NOT NULL,          -- ["선지1", "선지2", ...]
    correct_answer  SMALLINT NOT NULL,
    distractor_rationale TEXT,
    status          TEXT DEFAULT 'draft',    -- draft | approved | rejected
    created_at      TIMESTAMPTZ DEFAULT now()
);

-- 문항 피드백
CREATE TABLE question_feedback (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    question_id     UUID REFERENCES questions(id),
    score           SMALLINT CHECK (score BETWEEN 0 AND 10),
    comment         TEXT,
    corrections     JSONB,                   -- 사용자가 수정한 필드들
    created_at      TIMESTAMPTZ DEFAULT now()
);

-- 해설 (Phase 2)
CREATE TABLE explanations (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    question_id     UUID REFERENCES questions(id),
    template_id     UUID,                    -- 해설 양식 참조
    content         TEXT NOT NULL,
    status          TEXT DEFAULT 'draft',
    created_at      TIMESTAMPTZ DEFAULT now()
);

-- 해설 양식 템플릿 (Phase 2)
CREATE TABLE explanation_templates (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name            TEXT NOT NULL,
    structure       JSONB NOT NULL,          -- 양식 정의
    example         TEXT,                    -- 예시 해설
    created_at      TIMESTAMPTZ DEFAULT now()
);

Q: 왜 Claude API 우선이고 HuggingFace/RunPod은 optional인가?
A: Claude의 few-shot 능력이 충분히 강력하여, 피드백 데이터를 system prompt에
   주입하는 것만으로 "학습" 효과를 낼 수 있음. Fine-tuning은 데이터 500+개
   축적 후 비용/품질 트레이드오프를 검증한 뒤 진행해도 늦지 않음.

Q: 왜 verbatim 추출에 인덱스 방식을 쓰는가?
A: Claude에게 텍스트를 "출력"하게 하면 미세한 변형(환각) 위험이 있음.
   인덱스만 반환받고 백엔드에서 슬라이싱하면 원문 100% 보장.

Q: 왜 점수 체계를 6개 metric으로 나누는가?
A: 단일 점수보다 다차원 평가가 사용자 피드백을 더 정밀하게 반영함.
   "이 지문은 어휘는 좋은데 논리 구조가 약하다" 같은 피드백을 수치화 가능.
   가중치는 사용자 피드백 패턴에 따라 향후 자동 조정 가능.

Q: 왜 seed data를 JSONL로 관리하는가?
A: 1) 사용자가 텍스트 에디터에서 직접 작성/수정 가능 (DB 접근 불필요).
   2) Git으로 버전 관리 가능 (변경 이력 추적).
   3) 팀 작업 시 파일 단위로 분업 가능.
   4) DB 로드는 스크립트 한 번 실행으로 자동 처리.

Q: Seed data와 Feedback은 어떤 관계인가?
A: Seed = 변하지 않는 "정답지" (평가원이 실제 출제한 기출).
   Feedback = 사용자의 주관적 선호가 반영된 "진화하는 기준".
   초기엔 Seed가 100%, 피드백이 쌓이면 점진적으로 Feedback 비중 증가.
   하지만 Seed는 완전히 사라지지 않음 — anchor 역할로 기준선 유지.

Q: Pane 4를 왜 별도 엔진으로 분리하는가?
A: 해설 생성은 문항 생성과 다른 프롬프트/로직이 필요. 분리하면 각각
   독립적으로 개선 가능하고, 해설 양식이 확정되지 않은 현 상태에서도
   다른 기능 개발에 영향 없음.
